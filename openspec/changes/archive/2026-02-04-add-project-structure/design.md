## 上下文
项目定位为可复用的桌面基础设施框架，需要清晰区分启动入口、平台适配、核心能力与 UI 壳，以便后续演进与复用。

## 目标 / 非目标
- 目标：建立明确的目录分层与职责边界，减少新增代码放置的歧义。
- 目标：将关键架构约束显式化，避免跨层依赖。
- 非目标：本次不调整现有实现或引入新模块代码。

## 决策
- 采用固定的根目录分层（cmd/backend/core/ui-shell 等）描述职责与限制。
- 对关键二级目录给出最小但明确的职责说明。

## 风险 / 权衡
- 过度细化可能带来维护负担 → 采用“最小必要说明”，只覆盖关键目录。

## 迁移计划
- 仅更新项目约定与规范；不对现有代码做强制迁移。

## 待决问题
